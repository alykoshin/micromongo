The published user docs are a single hand-authored page at `docs/index.html`, served by GitHub Pages
from `/docs` on `master` (→ https://alykoshin.github.io/micromongo/). It runs the REAL engine in the
browser via `docs/micromongo.global.js` — a committed copy of the IIFE bundle (Pages serves only
committed files, so it can't reference gitignored `dist/`). `scripts/build-docs-site.js` copies the
fresh bundle + writes `.nojekyll`; `npm run build:docs` = build bundles + copy. `docs/` is NOT in
package.json `files[]` (won't bloat the npm tarball).

The page has a live playground and a functional-vs-Collection ops table computed against the engine.
The ops-table cards come from `docs/examples.generated.js` (a committed `window.__MM_EXAMPLES__`
global) — a CURATED 20-card showcase generated by `scripts/gen-docs-examples.js` from the canonical
example set `meta/mongo-examples/` (the `SHOWCASE` op-list picks one `docs:true` record per op). So the
playground can't drift from the tests — see [[unified-mongo-examples]]. **Gotcha:** playground
expressions are wrapped in `return (…)`, so multi-statement examples must be a single expression — wrap
in an IIFE (see the `update`/`collection` presets). The root README is deliberately slim and links here.

**Why committed-bundle over CDN:** works offline and on Pages with no npm publish needed. **How to
apply:** after an engine change that should reach the playground, run `npm run build:docs` (builds
bundles, copies the engine, regenerates `examples.generated.js`). See [[compat-notes-single-source]]
for the compat-table data pipeline and [[unified-mongo-examples]] for the example single-source.
